# 프로젝트 협업 규칙

4명이 같은 레포에서 서로 다른 AI 에이전트로 협업하기 위한 운영 기준입니다.

## 목적

- 팀원이 직접 코딩하기보다 AI에게 작업을 맡기는 방식을 전제로 합니다.
- 여러 명이 동시에 기능을 추가해도 공통 구조가 깨지지 않게 합니다.
- PR, 검토, 승인, 병합 기준을 정합니다.

## 기본값

### 역할 분담

- 팀원은 기획, 요구사항, 기준 결정, 결과 검토를 담당합니다.
- AI는 코드 수정, 문서 수정, 빌드 확인, 커밋, PR 생성을 담당합니다.
- 팀장은 PR 최종 승인과 `main` 병합을 담당합니다.
- 팀에서 확정한 기준은 AI가 담당 문서에 반영하고, 사람이 검토합니다.

### 작업 단위

작업은 아래 단위 중 하나로 구분합니다.

- 공고 도메인 (목록/상세/등록/지원)
- 프로필 도메인 (목록/상세/등록/연락처)
- 스토어/서비스·결제
- 커뮤니티/공지
- 마이페이지
- 관리자 화면
- 공통 컴포넌트/레이아웃
- 데이터 구조
- 이미지/에셋
- 문서 수정

### 구현 독립성

- 각 작업 단위는 가능한 한 **독립적으로 구현**합니다. 다른 팀원의 진행 상황을 몰라도 병합 시 깨지지 않는 것이 목표입니다.
- 비슷한 코드가 중복되는 것은 감수합니다. 병합 안전이 중복 제거보다 우선입니다.
- 예외(공유해야 하는 것): 데이터 이름(`docs/data-guide.md` 사전), 공통 컴포넌트(`components/ui`, `components/layout`), 디자인 토큰. **구현 코드는 독립, 이름과 토큰은 공유**가 기준입니다.
- 공유 파일(`components/ui/*`, `components/layout/*`, `lib/types.ts`, `lib/policy.ts`, `data/*`, `tailwind.config.ts`, `app/globals.css`) 수정은 팀 채널에 사전 공지하고, 가능하면 단독 PR로 분리해 먼저 병합합니다.

### 브랜치

- `main`은 항상 실행 가능한 상태로 유지합니다. 직접 push 금지, PR로만 병합합니다.
- 작업은 개인별 브랜치에서 진행합니다. 이름 형식: `이름/작업명` (예: `hyun/store-review`, `min/admin-members`).
- 브랜치는 항상 최신 `main`에서 분기하고, 작업 시작 전 `git pull`로 최신화합니다.
- 병합된 브랜치는 로컬과 원격에서 삭제합니다.

### 커밋

- 형식: `feat: 스토어 리뷰 작성 구현`, `fix:`, `docs:`, `style:`, `refactor:`, `chore:`
- 한 커밋에는 한 가지 목적만 담습니다.

### PR

- 모든 브랜치는 `main` 병합 전에 PR을 만듭니다. PR 생성과 본문 작성은 AI가 수행합니다.
- PR 1개 = 작업 단위 1개. 변경 500줄 이하 권장, 넘으면 쪼갭니다.
- PR 본문 필수 항목:
  1. 변경 요약 (무엇을, 왜)
  2. UI 변경 시 스크린샷 (PC 1280 / 모바일 390)
  3. 확인한 명령 (`npm run build`, `npm run lint` 결과)
  4. 데이터 사전(`docs/data-guide.md`) 변경 여부
  5. 사용한 AI 에이전트 표기 (예: `agent: codex`)
- PR은 긴 리뷰 문서가 아니라 `main` 진입 전 체크포인트로 사용합니다.

### 리뷰 통과 기준

- 빌드와 린트에 에러가 없어야 합니다.
- 역할 5종 값, 공통 컴포넌트, 데이터 구조를 이유 없이 바꾸지 않아야 합니다.
- 라우트 추가가 `docs/architecture.md`에 반영되어 있어야 합니다.
- 디자인이 `docs/design-system.md` 최소 기준을 크게 벗어나지 않아야 합니다.
- 검증 기준을 약화시키는 변경(lint 완화, `any` 남발)이 없어야 합니다.
- 디자인이 약간 어긋나는 정도는 후속 정리로 남길 수 있으나, 공통 구조를 깨는 변경은 후속으로 미루지 않습니다.

### 병합

- 승인 1명 이상 후 팀장만 병합합니다.
- 여러 PR을 한 번에 병합하지 않고, PR 단위로 하나씩 직렬 병합합니다.
- 병합 후 다음 PR은 최신 `main` 기준으로 충돌 여부를 확인합니다. 충돌은 해당 PR 작성자(의 AI)가 `git pull origin main`으로 해결하고 다시 push합니다.
- 데이터 구조, 디자인 토큰 변경이 포함된 PR은 추가 확인 후 병합합니다.
