## 1. 제품 정의

ScopeMatch는 제휴 수익을 키우려는 SNS 크리에이터가 숏폼 발행에 필요한 편집, 썸네일, 기획, 대본, 리서치 업무를 제휴 전환을 이해하는 외주 파트너에게 맡길 수 있게 돕는 매칭 플랫폼입니다.

크몽, 숨고처럼 범용 외주 인력을 검색하는 서비스가 아니라, 제휴 전환을 아는 숏폼 전문 파트너를 추천하고 작업 범위, 견적, 납기, 수정 기준을 표준화해 외주 실패 위험을 줄이는 것이 핵심입니다.

입력 워크시트 파일명과 일부 기존 문서에는 채용 플랫폼 용어가 남아 있지만, 제품 방향은 크리에이터 외주 매칭으로 정리합니다. 채용 플랫폼 잔재 용어는 개발 전 정리 대상입니다.

## 2. 핵심 고객

- 1번 고객: 인스타, 유튜브, 틱톡으로 제휴 수익을 키우려는 SNS 크리에이터
- 구체적 사용자: 발행량을 늘리기 위해 영상 편집, 썸네일, 기획을 외주로 맡기고 싶은 1인 운영자 김OO
- 집중하는 쪽: 수요 측 크리에이터

## 3. 문제 정의

제휴 수익을 키우려는 SNS 크리에이터 김OO는 숏폼 발행을 위해 편집, 기획을 외주로 맡기려 하지만, 제휴 전환을 이해하는 파트너를 가려낼 수 없고 작업 범위, 견적, 납기를 매번 새로 조율해야 해서 외주 실패 비용과 발행 지연의 고통을 겪습니다.

## 4. 핵심 가치

제휴 수익화에 필요한 반복 업무를 기획부터 전환 동선까지 아는 검증된 크리에이터, 전략가에게 빠르게 맡길 수 있게 합니다.

작업 범위, 견적, 납기, 수정 기준을 표준화해 외주 실패 위험을 줄입니다.

## 5. 차별점

- 범용 외주 검색이 아니라 제휴 전환을 아는 숏폼 전문 풀을 전제로 합니다.
- 단순히 사람 목록을 보여주는 것이 아니라 전환 나게 만들 사람을 추천합니다.
- 요청서 작성부터 검증된 파트너 추천까지 이어지는 매칭 흐름을 핵심 경험으로 둡니다.
- 전환 동선을 요청서의 필수 입력 구조로 다뤄, 예쁜 영상이 아니라 전환 목적에 맞는 작업을 맡기게 합니다.

## 6. MVP 핵심 기능

1. 요청서(Scope) 작성
   - 업무, 제휴 목표, 소재, 예산, 납기, 전환 동선, 완료 기준을 입력합니다.
2. 검증된 파트너 추천 및 비교
   - 제휴 성과, 포트폴리오, 견적 기준으로 크리에이터 또는 전략가 후보를 비교합니다.
3. 매칭 성사 및 작업 기준 합의
   - 작업 범위, 수정 기준, 견적, 수동 정산 상태를 확인합니다.

## 7. 전환 동선 입력 UX

전환 동선은 MVP 차별점이므로 단순 자유입력 한 칸으로 처리하지 않습니다.

- 입력 방식: 사용자가 직접 입력하되, 선택형 질문과 짧은 예시를 함께 제공합니다.
- 필수 입력: 제휴 목표, 제휴 상품/랜딩 URL, CTA 문구, 링크 배치 위치, 사용자가 최종으로 하길 원하는 행동
- 선택 입력: 참고 콘텐츠 URL, 타깃 시청자, 핵심 후킹 포인트
- 자동 추출: 채널 URL이나 콘텐츠 URL에서 AI가 자동 추출하는 기능은 MVP 후순위입니다.
- 미입력 처리: 전환 동선이 비어 있으면 추천 받기 전 보완 안내를 띄웁니다.

## 8. 콜드스타트 대응

MVP 초기에는 파트너 풀이 비어 있을 수 있으므로 “항상 3곳 추천”을 확정 흐름으로 두지 않습니다.

- 데모와 초기 검증에는 시드 파트너 데이터를 사용합니다.
- 초기 운영에서는 팀이 직접 영입한 소수 파트너를 우선 등록합니다.
- 추천 가능한 파트너가 3명 미만이면 가능한 수만 보여주고, 부족 사유와 수동 매칭 안내를 표시합니다.
- 추천 가능한 파트너가 0명이면 요청서를 접수하고 운영자가 직접 후보를 찾는 상태로 전환합니다.

## 9. 파트너 수급 전략

- 1안: 타겟 세분화 기반 수급
  - 편집몬 등 세분화된 외주 레퍼런스를 참고해 “숏폼 제휴 전환”에 맞는 좁은 파트너 풀을 먼저 만듭니다.
- 2안: 공개 후보 탐색
  - 공개 정보 기반으로 후보를 찾는 방식은 검토할 수 있습니다.
  - 자동 크롤링은 법적, 약관, 개인정보, 저작권 리스크가 있으므로 검토 전에는 구현하지 않습니다.
  - MVP에서는 수동 조사, 직접 영입, 동의 받은 파트너 데이터, 더미 데이터만 사용합니다.

## 10. 첫 사용 흐름

1. 사용자는 홈에서 외주 맡기기 또는 요청서 만들기 버튼으로 진입합니다.
2. 맡길 업무, 제휴 목표, 소재, 예산, 납기, 전환 동선, 완료 기준을 입력합니다.
3. 추천 받기 버튼을 누릅니다.
4. 추천 가능한 파트너를 비교합니다.
5. 파트너가 부족하면 수동 매칭 접수 상태를 확인합니다.
6. 파트너를 선택하고 작업 범위, 수정 기준, 견적, 수동 정산 상태를 확인합니다.

## 11. 지금은 제외할 기능

- 프로젝트 진행관리
- 마일스톤 자동화
- 산출물 리뷰 시스템
- 실시간 채팅 고도화
- 정교한 일정 관리 도구
- 자체 결제 에스크로 고도화
- 채널/콘텐츠 URL 기반 AI 자동 분석
- 약관 검토 전 자동 크롤링 수집

## 12. 가격 가설

- MVP 단일 검증 대상: 매칭 성사 수수료, 거래액의 10~20%
- 이후 검토: 고급 전략가 지명 매칭용 스카우트 건당 정액
- 이후 검토: 반복 외주 수요 확인 뒤 월정액 구독
- MVP 정산 방식: 실제 결제 연동 전에는 수동 정산 기록으로 검증합니다.

## 13. 성공 지표

- 매칭 성사 수수료 지불 의사 또는 실제 수수료 수납 여부

## 14. 화면 기준

### 화면 1. 요청서 작성

- 가치: 전환 나는 작업을 위해 무엇을 맡길지 정합니다.
- 행동: 맡길 업무, 소재, 예산, 납기, 전환 동선을 입력합니다.

- 가치: 전환을 아는 적합한 파트너를 찾습니다.

### 화면 3. 매칭 및 작업 기준 확인

- 가치: 맡기고 안전하게 마무리합니다.
- 행동: 파트너를 선택하고 작업, 견적, 정산 상태를 확인합니다.
- 구성요소: 작업 범위, 수정 기준, 견적 금액, 수수료율, 수동 정산 상태

## 16. 비기능 요건

- 결제: MVP에서는 자체 결제, 에스크로, PG 연동을 제외하고 수동 정산 기록으로 검증합니다.
- 개인정보: 크리에이터, 파트너 연락처와 포트폴리오 URL은 최소 수집하고, 공개/비공개 구분을 둡니다.
- 보안: Firebase SDK 값은 `.env`에만 저장하고 문서에는 기록하지 않습니다.
- 데이터: 실제 사용자 데이터가 들어가기 전 Firestore 보안 규칙을 검토합니다.
- 법무/약관: 자동 크롤링이나 외부 플랫폼 데이터 수집은 검토 전 구현하지 않습니다.

## 17. 온보딩 확정 사항

### 팀 구성

- 팀장: 김형언
- 팀장 역할: PR 최종 병합과 Hosting 배포 담당

### 디자인 방향

- 레퍼런스: Apaya
- 분위기: 미니멀하고 전문적인 업무 도구 UI
- 컬러: 화이트톤 배경, 블루 메인 컬러, 라임 서브 컬러
- 꼭 넣을 느낌: 깔끔함, 신뢰감, 직관적 사용 흐름
- 피할 느낌: 너무 화려함, 쇼핑몰 느낌, 장식 과다

### AI 작업 규칙

- AI는 요청받은 코드 수정, 문서 수정, 실행 확인, 빌드 확인, 커밋, PR 생성을 수행합니다.
- 파일 삭제, 대규모 이동, 의존성 추가, 공통 구조 변경은 먼저 확인을 받습니다.
- 공통 구조는 SPA 구조와 역할 기반 화면 분기를 의미합니다.

### 협업 기준

- 개인별 작업 브랜치에서 작업합니다.
- 브랜치 이름은 `student/이름/작업명` 형식을 사용합니다.
- PR은 필수입니다.
- AI가 PR을 작성하고 리뷰하며, 팀장이 최종 승인 후 `main`에 병합합니다.
- PR은 하나씩 직렬 병합합니다.

### Firebase 기준

- Firebase SDK 값은 `.env`에만 저장하고 문서에는 기록하지 않습니다.
- 공유용 변수명은 `.env.example`에 값 없이 둡니다.
- 지금은 Hosting 중심으로 사용합니다.
- Firestore, Storage는 필요 시 사용합니다.
- 실제 사용자 데이터가 들어가기 전 Firestore 보안 규칙을 검토합니다.
