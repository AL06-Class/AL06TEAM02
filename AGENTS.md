# AGENTS.md — 촬영몬 협업 규약

이 문서는 촬영몬 구현 레포에서 **모든 AI 에이전트와 팀원이 반드시 지켜야 할 공통 규칙**입니다.

> **사용 안내**: 구현 레포를 만들 때 이 파일을 레포 루트에 `AGENTS.md`로 둡니다.
> 기존 기획 단계 `AGENTS.md`(이미지 프리셋/기획 지침)는 `docs/planning-guide.md`로 옮깁니다.

## 목적

- 4명이 서로 다른 AI 에이전트로 작업해도 결과물이 흔들리지 않게 합니다.
- 설계 문서 9종이 이미 완성되어 있으므로, **구현은 문서를 따르고 문서에 없는 것은 임의로 정하지 않습니다.**
- AI가 공통 구조(라우팅, 역할, 데이터 스키마, 디자인 토큰)를 임의로 바꾸지 않게 합니다.

## 에이전트 공통 원칙 (어떤 AI를 쓰든 동일)

- 이 문서가 이 레포의 **최상위 실행 규칙**입니다. 각자의 에이전트 설정 파일이 이 문서와 충돌하면 이 문서를 우선합니다.
- 에이전트별 연결 방법:
  - **Codex / Cursor**: 루트 `AGENTS.md`를 자동 인식하므로 추가 설정 불필요
  - **Claude Code**: 레포 `CLAUDE.md` 첫 줄에 `@AGENTS.md` 참조를 넣어 이 문서를 로드
  - **Gemini CLI**: `GEMINI.md`에 동일하게 참조
- 어떤 에이전트든 세션 시작 시 이 문서 + 작업 성격에 맞는 담당 문서(아래 라우팅 표)를 먼저 읽습니다.
- AI가 만든 결과는 **작업자 본인이 검토한 뒤** 커밋합니다. "AI가 만들었으니 모른다"는 리뷰에서 허용되지 않습니다.

## 문서 라우팅

모든 문서를 매번 읽지 않습니다. 작업 성격에 맞는 문서만 확인합니다.

| 작업 성격 | 담당 문서 |
| --- | --- |
| 기능 요구/정책이 애매할 때 | `PRD.MD` (최우선 기준) |
| 라우트 추가/변경, 권한, 페이지 흐름 | `architecture.md` (3장 라우팅, 4~5장 권한/게이트, 6장 플로우) |
| 화면 배치, 어떤 요소가 들어가는지 | `wireframe.md` / 마이페이지는 `mypage-design.md` / 관리자는 `admin-design.md` |
| 색상, 간격, 버튼/배지/카드 스펙 | `ui-components.md` (토큰·컴포넌트), 배경 원칙은 `design.md` |
| 데이터 필드, 상태값, 샘플 데이터 | `sample-data.md` (**공통 이름 사전**) |
| 이미지 파일, alt, 프롬프트 | `image-presets.md` |
| 구현 순서, 완료 기준, 미확정 정책 가정값 | `handoff.md` |

## 데이터 변경 경로 (중요)

- 새 필드, 상태값, 카테고리, 역할 값이 필요하면 **코드보다 먼저 `sample-data.md`의 스키마를 수정**하고 PR에 포함합니다.
- 기존 이름이 있으면 재사용합니다. 같은 개념에 두 번째 이름을 만들지 않습니다 (예: `isPremium`이 있는데 `premium` 추가 금지).
- `lib/types.ts`는 `sample-data.md` 2장과 항상 일치해야 합니다. 어긋나면 문서를 먼저 고칩니다.

## 유지해야 할 공통 구조

- Next.js App Router + TypeScript + Tailwind. **백엔드/DB/외부 API/의존성 추가는 팀 합의 전 금지.**
- 역할 5종 `guest / personal / company-unverified / company-verified / admin` — 이름과 개수를 바꾸지 않습니다.
- 인증은 `lib/auth-context.tsx` 역할 스위처 시뮬레이션 방식을 유지합니다. 실제 로그인 구현 금지(2차 합의 사항).
- 새 라우트는 `architecture.md` 3장에 먼저 추가한 뒤 구현합니다.
- 색상/간격/그림자 토큰 임의 추가 금지 — 필요하면 `ui-components.md` 1장 수정 PR을 먼저 냅니다.
- 미확정 정책 값(열람권 1일 기준 등)은 `lib/policy.ts` 상수만 사용하고 하드코딩하지 않습니다 (`handoff.md` 5장).
- 쓰기 동작(지원/스크랩/승인)은 localStorage 방식을 유지합니다.

## 절대 금지 (기획 단계 규칙 승계)

- 편집몬(editmon.com)의 로고, 이미지, 문구, 게시글 원문, 사업자 정보를 복제하지 않습니다.
- 이미지 안에 텍스트/로고/워터마크를 넣지 않습니다. 이미지는 `image-presets.md` 프리셋 기준으로만 생성합니다.
- 검증을 통과시키려고 lint 규칙 완화, 타입 `any` 남발, 빌드 에러 우회 설정을 하지 않습니다. **검증 기준을 약화시키는 변경은 그 자체로 리뷰 반려 사유입니다.**
- 팀원의 변경사항을 임의로 되돌리거나(`git checkout --`, force push) 삭제하지 않습니다.

## 브랜치 / 커밋 / PR

### 브랜치

- `main`은 보호 브랜치 — 직접 push 금지, PR로만 병합
- 브랜치 이름: `feat/영역-내용`, `fix/영역-내용`, `docs/내용`
  - 예: `feat/jobs-list`, `feat/mypage-verification`, `fix/header-mobile-tab`
- 브랜치는 항상 최신 `main`에서 분기하고, 작업 시작 전 `git pull`로 최신화합니다.

### 커밋

- 형식: `feat: 공고 목록 필터 구현`, `fix:`, `docs:`, `style:`, `refactor:`, `chore:`
- 한 커밋에는 한 가지 목적만 담습니다. AI가 만든 커밋 메시지도 이 형식을 따릅니다.

### PR

- PR 1개 = 화면 또는 기능 1개. 변경 500줄 이하 권장 — 넘으면 쪼갭니다.
- PR 본문 필수 항목:
  1. 변경 내용 요약 (무엇을, 왜)
  2. **UI 변경 시 스크린샷 필수** — PC 1280px + 모바일 390px
  3. 체크: `npm run build` 통과 / `npm run lint` 통과 / 관련 문서와 일치
  4. 사용한 AI 에이전트 표기 (예: `agent: codex`) — 문제 추적용
- 리뷰: 팀원 1명 이상 승인 → **팀장이 최종 병합**. AI는 리뷰 의견은 낼 수 있으나 병합 주체가 되지 않습니다.
- 리뷰 우선 확인: 공통 구조 훼손 여부 → 빌드/린트 → 문서 기준 위반 → 코드 품질 순서

### 충돌 방지

- **공유 파일**(`components/ui/*`, `components/layout/*`, `lib/types.ts`, `lib/policy.ts`, `data/*`, `tailwind.config`, 전역 CSS)을 수정할 때는 작업 시작 전에 팀 채널에 먼저 공지합니다.
- 공유 파일 변경은 가능하면 단독 PR로 분리해 먼저 병합하고, 기능 PR은 그 위에서 작업합니다.
- 같은 파일을 두 명이 동시에 작업하지 않도록 이슈/작업 보드에 담당을 먼저 적습니다.

## 작업 절차 (모든 작업 공통)

1. `main` 최신화 → 브랜치 생성
2. 이 문서 + 라우팅 표의 담당 문서 확인
3. 기존 컴포넌트/유틸 재사용 우선 — 새로 만들기 전에 `components/` 검색
4. 구현 → `npm run build`와 `npm run lint` 통과 확인
5. UI 변경이면 PC/모바일 스크린샷 캡처
6. 요청 범위 밖 리팩터링 금지 — 발견한 문제는 고치지 말고 이슈로 남깁니다
7. PR 생성 → 리뷰 → 팀장 병합

## 보고 기준 (AI 응답 공통)

- 결론부터 짧게. 변경한 파일과 확인한 명령만 공유합니다.
- 실패한 검증(빌드 에러, 미확인 사항)은 숨기지 않고 그대로 보고합니다.
- 요청 없는 코드를 본문에 길게 출력하지 않습니다.
- 사용자 판단이 문서 기준과 다르면 동의하지 말고 근거 문서를 짚어줍니다.

## 문서와 코드가 충돌할 때

1. 기능 요구: `PRD.MD` 우선
2. 화면/디자인: `wireframe.md` · `ui-components.md` 우선
3. 문서끼리 충돌하거나 문서가 틀렸으면 — 코드로 우회하지 말고 **문서 수정 PR을 먼저** 올려 팀 합의를 받습니다.
4. 합의된 변경은 이 문서의 `최종 결정`과 `변경 이력`에 AI가 반영하고 사람이 검토합니다.
