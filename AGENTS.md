# AGENTS.md

이 문서는 이 레포에서 AI가 작업할 때 반드시 지켜야 할 실행 규칙입니다.

## 목적

- 4명이 서로 다른 AI 에이전트(Codex, Cursor, Claude Code, Gemini 등)로 작업해도 결과물이 흔들리지 않게 합니다.
- AI가 임의 판단으로 공통 구조, 데이터 구조, 디자인 기준을 바꾸지 않게 합니다.
- 협업 운영 기준은 `docs/project-rules.md`에 두고, 이 문서는 AI 실행 규칙만 다룹니다.

## 기본값

### 문서 기준

- 협업, 브랜치, PR, 승인, 병합 기준은 `docs/project-rules.md`를 따릅니다.
- 역할, 데이터 이름, 상태값, 샘플 데이터 기준은 `docs/data-guide.md`를 따릅니다.
- 화면 디자인 기준은 `docs/design-system.md`를 따릅니다.
- 실행, 빌드, 터널 공유, 배포 기준은 `docs/deploy-guide.md`를 따릅니다.
- 기능 요구사항이 애매하면 `docs/PRD.MD`가 최우선 기준입니다.
- 라우트, 권한, 페이지 플로우는 `docs/architecture.md`를 따릅니다.

### 문서 확인 방식

- 모든 작업에서 모든 문서를 매번 읽지 않습니다.
- 작업 전에는 `README.md`와 이 문서를 먼저 확인하고, 작업 성격에 맞는 담당 문서만 추가로 확인합니다.
- 데이터 이름, 필드, 상태값, 역할 값이 관련되면 `docs/data-guide.md`를 확인합니다.
- 화면 톤, 레이아웃, 컴포넌트, 반응형이 관련되면 `docs/design-system.md`를 확인합니다.
- 브랜치, PR, 병합, 승인, 작업 순서가 관련되면 `docs/project-rules.md`를 확인합니다.
- 실행, 배포, 외부 공유가 관련되면 `docs/deploy-guide.md`를 확인합니다.
- 2차 화면 구현 시에만 `docs/reference/wireframe.md`, `docs/reference/mypage-design.md`, `docs/reference/admin-design.md`를 참조합니다.
- 이미지 생성 시에만 `docs/reference/image-presets.md`를 참조합니다.

### 에이전트 연결

- Codex / Cursor: 루트 `AGENTS.md` 자동 인식.
- Claude Code: `CLAUDE.md`가 이 문서를 참조.
- Gemini CLI: `GEMINI.md`에 `@AGENTS.md` 참조 추가.
- 각자의 에이전트 설정이 이 문서와 충돌하면 이 문서를 우선합니다.

### AI 역할

- AI는 코드 수정, 문서 수정, 실행 확인, 빌드 확인, 커밋, PR 생성을 수행합니다.
- 팀원은 요구사항 결정, 기준 합의, 결과 검토를 담당합니다.
- 팀 결정사항은 AI가 담당 문서의 `최종 결정`과 `변경 이력`에 반영하고, 사람이 검토합니다.
- 팀장은 PR 최종 병합을 담당합니다. AI는 병합 주체가 되지 않습니다.
- AI는 정해지지 않은 기준을 임의로 확정하지 않습니다.

### 작업 전 확인

- 현재 Git 상태를 확인하고, 팀원의 변경사항을 임의로 되돌리지 않습니다.
- 기존 파일 구조와 작성 방식을 먼저 확인합니다.
- 새로 만들기 전에 `components/`와 `lib/`에서 재사용할 것을 먼저 찾습니다.
- 데이터 이름을 새로 만들거나 바꿔야 하면 코드보다 먼저 `docs/data-guide.md`의 공통 이름 사전을 거칩니다.

### 유지해야 할 구조

- Next.js App Router + TypeScript + Tailwind. 백엔드/DB/외부 API/의존성 추가는 팀 합의 전 금지.
- 역할 5종 `guest | personal | company-unverified | company-verified | admin` — 이름과 개수를 바꾸지 않습니다.
- 인증은 `lib/auth-context.tsx` 역할 스위처 시뮬레이션을 유지합니다. 실제 로그인 구현은 팀 합의 사항입니다.
- 새 라우트는 `docs/architecture.md` 3장에 먼저 추가한 뒤 구현합니다.
- 미확정 정책 값은 `lib/policy.ts` 상수만 사용하고 하드코딩하지 않습니다.
- 쓰기 동작(지원/스크랩/승인/결제)은 localStorage 방식을 유지합니다.

### 수정 기준

- 요청 범위 밖의 리팩터링은 하지 않습니다.
- 파일 삭제, 대규모 이동, 의존성 추가는 먼저 이유가 명확해야 하고 팀 확인이 필요합니다.
- 기존 컴포넌트, 스타일, 데이터 구조를 우선 재사용합니다.
- 화면마다 별도 디자인 톤을 만들지 않습니다.
- 문서 기준과 다르게 구현해야 하면 문서 수정 여부를 먼저 검토합니다.

### 절대 금지

- 편집몬(editmon.com)의 로고, 이미지, 문구, 게시글 원문, 사업자 정보 복제 금지.
- 이미지 안에 텍스트/로고/워터마크 삽입 금지. 이미지는 `docs/reference/image-presets.md` 기준으로만 생성.
- 검증 통과를 위한 lint 규칙 완화, 타입 `any` 남발, 빌드 에러 우회 설정 금지 — 그 자체로 리뷰 반려 사유입니다.
- 팀원의 변경사항을 임의로 되돌리는 행위(`git checkout --`, force push) 금지.

### 보고 기준

- 작업 후 변경한 파일과 확인한 명령을 짧게 공유합니다.
- 실패한 검증(빌드 에러, 미확인 사항)은 숨기지 않고 그대로 보고합니다.
- 선택지가 있으면 수정 전 상태와 수정 후 결과를 구분해서 설명합니다.

### 응답 기준

- 결론부터 짧게 말합니다.
- 요청 없는 코드를 본문에 길게 출력하지 않습니다.
- 사용자 판단이 문서 기준과 다르면 동의하지 말고 근거 문서를 짚어줍니다.
